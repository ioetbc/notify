# Expo Push Integration — Technical Decisions

This document captures the architectural decisions and learnings for integrating with Expo's Push Notification service.

---

## Notification Delivery Flow

```
Your Server → Expo API → APNs (iOS) / FCM (Android) → User's Device
```

- Expo acts as a unified proxy over Apple's APNs and Google's FCM
- Expo handles platform-specific routing based on the push token
- We never deal with APNs certificates or FCM credentials directly
- Delivery cost is effectively zero (Expo, APNs, and FCM are all free)

---

## Push Tokens

### What they are
- A unique device identifier generated by Expo when the user grants notification permission
- Format: `ExponentPushToken[xxxxxx]`
- Tied to the specific app and device

### Important behaviors
- **Tokens can change** — they don't have a formal expiry, but become invalid when:
  - User reinstalls the app
  - User restores from backup to a new device
  - OS decides to rotate it (rare)
- **One user can have multiple tokens** — if they use multiple devices (e.g., phone and tablet, or shared accounts)
- **Relationship**: `user_id` → `push_token` is **one-to-many**

### Token registration
- Token is created when the app calls `Notifications.getExpoPushTokenAsync()`
- This only succeeds **after** the user accepts the native permission prompt
- On iOS, you only get **one chance** to show this prompt — if declined, user must manually enable in Settings
- Customer apps should ask for permission at an optimal moment (when user is "warm")

## API Integration

### Sending notifications

```javascript
const message = {
  to: expoPushToken,
  sound: 'default',
  title: 'Your order shipped!',
  body: 'Track your delivery',
  data: { screen: 'OrderDetails', orderId: '12345' },
};

await fetch('https://exp.host/--/api/v2/push/send', {
  method: 'POST',
  headers: {
    Accept: 'application/json',
    'Accept-encoding': 'gzip, deflate',
    'Content-Type': 'application/json',
  },
  body: JSON.stringify(message),
});
```

### Batching
- Expo allows up to **100 notifications per API request**
- For large sends (e.g., 10,000 users), batch into groups of 100
- Each batch becomes a single queue message for processing

## Tickets and Receipts

Expo uses a two-phase async delivery confirmation system:

### Phase 1: Tickets
- When you call the push API, Expo immediately returns a **ticket** with an ID
- The ticket only confirms Expo accepted the request — **not** that it was delivered
- Store the ticket ID for later receipt checking

### Phase 2: Receipts
- Call `POST /push/getReceipts` with ticket IDs to get delivery status
- **Wait at least 15 minutes** before checking — APNs/FCM need time to report back
- Receipts are available for **~24 hours**, then discarded

### Receipt responses
| Status | Meaning | Action |
|--------|---------|--------|
| `ok` | Delivered successfully | Mark as SUCCESS |
| `DeviceNotRegistered` | Token is dead | Mark token as inactive, stop sending |
| `MessageTooBig` | Payload exceeded limit | Log error, notify customer |
| `InvalidCredentials` | Auth issue | System-level alert |

## Queue Architecture

### Sending flow
1. Journey engine identifies users to notify
2. Server batches into groups of 100 tokens
3. Each batch goes onto **send queue** (SQS or similar)
4. Lambda/worker picks up batch, calls Expo API, receives tickets
5. Tickets go onto **receipt queue** with 15-minute delay

### Receipt checking flow
1. Receipt queue has visibility timeout / delay of 15 minutes
2. Worker picks up ticket batch after delay
3. Calls Expo's `/push/getReceipts` endpoint
4. Updates notification status in database
5. Marks invalid tokens as inactive

## Data Model Implications

### Users / Push Tokens
- One user can have multiple tokens (multi-device)
- Each token has its own `active/inactive` status
- When `/users/register` is called, **add** token (don't replace)
- Upsert if same token is sent again (no duplicates)

### Notifications Sent Table
- **One row per token/delivery attempt** (not per user)
- If user has 3 devices, sending 1 notification = 3 rows
- Each row tracks: ticket_id, status (PENDING/SUCCESS/FAILED), error_message
- Allows reporting: "This notification reached 2 of 3 devices"

### Handling DeviceNotRegistered
1. Mark the token as `inactive` in database
2. Journey engine skips inactive tokens on future sends
3. Surface aggregate stats to customer: "X% of tokens are stale"
4. When app re-registers with fresh token, overwrite and mark active

## Authentication

### Decision: Single shared Expo access token (Option A)

**Rationale:**
- Simpler onboarding — customers don't need their own Expo account
- Expo push tokens are tied to the customer's app, not our account
- Notifications still "come from" the correct app based on the token itself

**Trade-offs to monitor:**
- Shared rate limits across all customers
- Single point of failure if token is revoked
- May need to revisit at scale

**Future option:** Per-customer Expo tokens if isolation becomes necessary

## Notification Content

### What we control (per notification)
- `title` — bold heading text
- `body` — message content
- `data` — custom payload for deep linking
- `sound` — default or custom
- `badge` — app icon badge count

### What we don't control
- **App icon** — comes from installed app (customer's app.json)
- **App name** — comes from installed app
- These are baked into the customer's app, nothing to configure on our side

---

## Deep Linking

### Structure
The `data` field carries deep link information:
```javascript
data: {
  screen: 'OrderDetails',
  orderId: '12345'
}
```

### UI form fields
- **screen** — the screen/route name in the customer's app
- **id** — dynamic identifier (often from event properties)

### Customer responsibility
The customer's app must:
1. Listen for notification taps (Expo notification response listener)
2. Read the `data` payload
3. Navigate to the appropriate screen

We send the data; they handle the navigation.

## Dynamic Templating

### Syntax
Handlebars-style templates for personalizing notifications:
```
Title: "Your order {{event.properties.productName}} shipped!"
Deep link ID: {{event.properties.orderId}}
```

### Event property schema
- **Discovery approach** (like Loops.so): Learn schema from events we've received
- When marketer types `{{event.properties.`, autocomplete shows properties we've seen for that event type
- Trade-off: New properties aren't available until at least one event with that property is received

### Fallback values
- Prompt marketer to set fallback when inserting a variable
- Show warning on journey canvas if any notification has tags without fallbacks
- Example: `{{event.properties.productName | default: "your item"}}`
- Prevents broken notifications when properties are missing
