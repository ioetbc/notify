Push Notification Platform — System Overview

What We're Building
A B2B SaaS platform that allows businesses to send automated, event-driven push notifications to their mobile app users. Think of it as "Loops for push notifications" — a beautifully designed, developer-friendly tool that lets small-to-medium sized businesses create sophisticated notification flows without enterprise complexity or pricing.
The platform sits between a business's app and their end users' devices. Businesses integrate once, and then their marketing or product teams can build automated notification journeys through a visual canvas UI — no ongoing developer involvement required after the initial setup.

Target Customer

Small-to-medium consumer app companies with between 1,000 and 500,000 monthly active users
Apps built with Expo or React Native — this is our initial platform constraint and deliberate wedge
Teams currently using OneSignal but frustrated by feature gating on the free tier
Businesses where marketing teams need to create campaigns without going back to developers every time


Core Concepts
Customer — the business that pays us and integrates our platform into their app.
End User — the consumer who uses our customer's app and receives push notifications.
Push Token — a unique device identifier generated by Expo when an end user grants notification permissions. This is the address we use to deliver notifications to a specific device.
Event — something meaningful that happens in the customer's app or backend, such as a user signing up, completing a purchase, or going inactive. Events are the triggers that drive all journey logic.
Journey — a visual workflow built by the customer in our canvas UI. A journey listens for a trigger event and then executes a sequence of steps — waits, conditions, branches, and notification sends — as end users flow through it.

How the Integration Works
Step 1 — Token Registration
When an end user opens the customer's app for the first time and grants notification permissions, Expo generates a push token for that device. The customer's app sends this token to our API along with a user ID from their own system. We store the mapping of user ID to push token.
This is a single API call:
POST /users/register
{
  "user_id": "abc123",
  "push_token": "ExponentPushToken[xxxxxx]"
}
Step 2 — Event Tracking
Whenever something meaningful happens in the customer's app or backend, they fire an event at our API. This tells us what the user did and when.
POST /events
{
  "user_id": "abc123",
  "event": "purchase_completed",
  "properties": {
    "plan": "pro",
    "amount": 49.00
  }
}
Events can be fired from the mobile app itself or from the customer's backend server — wherever the action originates.
Step 3 — Journey Execution
Our platform receives the event, looks up which journeys are listening for that trigger, and begins processing any affected users through those journeys. The journey engine handles all timing, branching, and condition logic. When a notification step is reached, we call Expo's Push API to deliver the notification to the user's device.
The customer's developers never need to think about this layer. They fire events. We handle everything else.

The Journey Canvas
The core product surface is a visual infinite canvas where customers drag and drop nodes to build notification workflows. The canvas is the main differentiator — it needs to feel as polished as a Shopify Flow or Linear, not like a legacy enterprise tool.
Node types include:

Trigger — the entry point, listens for a specific event such as "user_signed_up"
Wait — pauses the journey for a set duration, such as 2 days
Condition / If-Else — branches based on whether another event has occurred, or based on user properties. For example: "has the user completed a purchase?"
Send Notification — delivers a push notification with a title, body, and optional deep link

Notification Delivery Architecture
We use Expo's Push Notification Service as our delivery layer. This is a deliberate architectural choice.
Expo acts as a unified abstraction over Apple's APNs (iOS) and Google's FCM (Android). When we call Expo's API with a push token and a message, Expo handles routing the notification to the correct platform and device. This means we never deal with APNs certificates, FCM credentials, or platform-specific delivery logic directly.
Critically, Expo charges nothing for notification delivery. APNs and FCM are also both free. Our delivery cost per notification is effectively zero, which gives us exceptional gross margins — estimated at 85 to 95%.
The constraint: this only works for apps built with Expo or React Native. Apps built natively in Swift or Kotlin cannot use Expo push tokens. This is an intentional v1 limitation that defines our target customer.

Data Model (High Level)
Customers — the businesses using our platform, with API keys for authentication.
Users — end users within each customer's app, identified by the customer's own user ID and mapped to one or more push tokens.
Events — a log of every event received per user, with timestamp and any properties attached.
Journeys — the workflow definitions built in the canvas, including all nodes, edges, and configuration.
Journey Enrollments — a record of which users are currently active in which journeys, and which step they are at.
Notifications Sent — a log of every notification dispatched, with delivery status pulled back from Expo's receipt API.

What We Are Not Building (v1)

Native iOS or Android SDK support — Expo only for now
Direct database connections — we are purely event-driven, customers push events to us
Email, SMS, or in-app messaging — push notifications only
Self-serve native APNs or FCM credential management
Enterprise features such as SSO, audit logs, or dedicated infrastructure

Competitive Context
OneSignal is the dominant SMB player with over 2 million developer users. Their weaknesses are aggressive feature gating on the free tier, a journey builder that limits free users to 1 journey with only 2 steps, and a product that was built for developers rather than marketers.
Pushbase is the closest direct competitor — an Expo-specific push notification service — but it does not have a visual journey canvas and is much more limited in automation capabilities.
Braze and Airship are enterprise-only, starting at $60,000 per year, and are not competing for our customer.
Our positioning: the first genuinely design-led, marketer-friendly push notification platform built specifically for the Expo and React Native ecosystem, with generous pricing and no feature gating.
